### T1.3 Security finding worth a paragraph in the security chapter
`EXPO_PUBLIC_*` env vars are inlined into the JS bundle at build time. Anyone who pulls the APK can extract `EXPO_PUBLIC_BRACELET_MASTER_KEY`. From there, every per-bracelet HMAC key is derivable. The thesis defends the design partly on chip integrity via HMAC, so the master key sitting in the mobile bundle is a load-bearing weakness that the security chapter (currently a stub) must acknowledge and frame as MVP-only.

A second concern: there is a hardcoded fallback dev key `00112233...eeff` in `softpos/lib/config.ts:10`. If the env var is missing in production, that becomes the active key. Either remove the fallback or fail loudly if the env var is unset.

### T1.4 Abstract promises an "assessment" that the body does not deliver
`docs/main.tex:41`: "We assess the architecture against what a real festival actually needs." There is no evaluation chapter, no requirements list, no measurements (no tap latency, no throughput, no battery numbers), no benchmark, no simulated festival run. Ch7's trade-off discussion is honest reasoning but not assessment with criteria. Either soften the abstract claim or add a short "Evaluation against festival requirements" section with a concrete checklist (peak TPS, acceptable outage duration, reconciliation correctness verdict).

- `chapter1_introduction.tex:49` (O6) is "evaluate against festival requirements, deferred to ch7" — but ch7 does not deliver an evaluation (see T1.4). Either rewrite O6 as "discuss trade-offs" or actually deliver the evaluation.

### T2.4 Ch2 scope leakage

### T2.5 Ch2 §2.3.3 strawmans vector clocks
`chapter2.tex:168` claims vector clocks are "necessary in systems like high-frequency trading platforms". HFT mostly uses synchronized physical clocks (PTP), not vector clocks; vector clocks track *causal ordering*, not microsecond physical time. The honest argument the thesis wants to make is "we don't need causal ordering across terminals because each balance has at most one writer per lane." That argument is already in ch4 and is much stronger. Rewrite §2.3.3 around it, drop the HFT line.

### T2.6 EMV is misclassified
EMV is the canonical open-loop standard, but `chapter2.tex:23` files it under "Closed-Loop Payment Systems". Move §2.1.3 to its own short section "Offline authorization in open-loop systems" between §2.1 and §2.2.

### T2.7 Ch5 missing things an implementation chapter needs
- Project layout (where the files live)
- Libraries chosen and why (NestJS, class-validator, `@noble/hashes`, `expo-file-system`, `@react-native-community/netinfo`, Better Auth, Drizzle)
- Auth flow (Better Auth sessions with `bearer` plugin — the chapter says "JWT" wrongly, no JWT here)
- Webhook raw-body trick (`req.rawBody` for Stripe signature verification)
- NFC session serialization (`withSession` chains all NFC reads through one promise — real gotcha worth a paragraph)
- Auto-sync on connectivity change (`@react-native-community/netinfo` triggers sync)
- Auto-sync without re-tapping (`runAutoSyncForBracelet` syncs from the snapshot, does not write back to the chip)
- Tests that exist (`reconciliation.e2e-spec.ts`, `bracelets.e2e-spec.ts`, `bracelet-token.spec.ts`)
- DB constraints used as safety nets (`event_bracelets_balance_check (balance >= 0)`, partial unique indexes)
- The hardcoded reviewer accounts in `auth.ts` that must be stripped before deployment

### T2.8 Security chapter rewrite should be threat-driven
The current skeleton has generic section headings (Auth, Wallet Integrity, Offline Trust, Stripe, Data Protection). For this architecture the right structure is:
1. Threat model (assets, actors, trust boundaries)
2. Chip integrity (HMAC, replay-after-topup, cloning, what counter regression detects)
3. **Cryptographic key management** (HKDF, master key on terminal, EXPO_PUBLIC bundle problem, HSM/DESFire upgrade path) — load-bearing
4. Server-side integrity (cite ch4/ch5 for serializable + single-writer-per-lane)
5. Auth (Better Auth sessions, OTP, reviewer-account caveat, role guard vs per-event membership gap)
6. Stripe (webhook signature, PCI scope reduction)
7. Vendor terminal trust (whitelisting, root-extraction risk)
8. Data protection / GDPR
9. DoS / availability
10. Residual risks
