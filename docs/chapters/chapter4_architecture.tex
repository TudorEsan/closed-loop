\chapter{System Architecture}
\label{chap:architecture}

This chapter goes through the architecture of the closed-loop payment system. We start with the big picture, explain why we chose this approach, and then go into each component. The idea is to keep things simple: one server, one app for everyone, and reliable network whenever possible. If the network goes down, the balance cached on the bracelet is what decides whether a payment is approved.

\section{High-level overview}
\label{sec:high-level-overview}

Most existing festival payment systems need a technical team on-site to set up local servers and a private network for the duration of the event. As discussed in Section~\ref{subsec:campus-cards}, festivals like Untold bring dedicated servers to the venue because the balance lives on a remote server and every transaction needs a network round-trip.

Our system takes a different path. We aim to always have reliable internet using Starlink as the primary uplink, paired with a redundant network topology. The wristband-as-cache fallback exists for residual edge cases, but the goal is to never need it. This means there is nothing to deploy at the venue: no local servers, no private network, no on-site IT team. The organizer creates an event from the admin dashboard, configures it, approves vendors, and distributes wristbands. The server runs in the cloud the same way it runs for every other festival on the platform.

The system has four main components: a backend server, a mobile application for attendees and vendors, a web-based admin dashboard, and NFC wristbands that carry the token balance for each attendee. The server is a single multi-tenant service that manages all festivals from one deployment. The mobile app is what attendees and vendors interact with: attendees see their balance and history, vendors use it as a soft point-of-sale terminal. Admins manage the event through a separate web dashboard. The wristband is where the balance actually lives during the event.

\subsection{The hybrid reconciliation model}
\label{subsec:hybrid-overview}

The core architectural decision is how the wristband and the server share responsibility for the balance. 

The state is split across the two sides. Each wristband stores three fields per event: \texttt{balance}, \texttt{debit\_counter} (incremented on every chip-side payment), and \texttt{credit\_counter\_seen} (the highest server credit sequence the chip has acknowledged). The server keeps the mirror image: \texttt{balance}, \texttt{debit\_counter\_seen}, and \texttt{credit\_counter}. Balances are event-scoped, so the same person attending two festivals has two independent wallets.

For this thesis demo we use a \textbf{plain NTAG-class NFC chip} without a secure element. The hybrid model does not depend on chip-side cryptography: it only needs the chip to store a few fields, increment a counter atomically, and keep what was last written to it. The security implications and the upgrade path to authenticated chips are discussed in Chapter~\ref{chap:security}.

\section{Balance reconciliation model}
\label{sec:reconciliation}

A closed-loop payment system with offline-capable terminals has two writers that can mutate the balance: the server (top-ups) and the chip (vendor payments). During a network partition both can act on their own, and once connectivity is back the system has to converge on a correct final value. Our hybrid model handles this by keeping the two writers in separate lanes that never overlap.

\subsection{State on each side}
\label{subsec:reconciliation-state}

The wristband stores three reconciliation fields per event, plus identity fields written at provisioning:
\begin{itemize}
	\item \texttt{balance}: cached balance in minor units, what the chip can spend from when offline.
	\item \texttt{debit\_counter}: monotonic counter, incremented by one on every chip-side debit. The chip is the only writer.
	\item \texttt{credit\_counter\_seen}: the highest server-side credit sequence number the chip has acknowledged.
	\item identity fields \texttt{bracelet\_uid} and \texttt{event\_id}, written once at provisioning.
\end{itemize}

The server stores the mirror image, per wristband per event:
\begin{itemize}
	\item \texttt{balance}: the canonical balance.
	\item \texttt{credit\_counter}: monotonic counter, incremented by one on every server-side credit. The server is the only writer.
	\item \texttt{debit\_counter\_seen}: the highest \texttt{debit\_counter} value the server has reconciled from any terminal.
\end{itemize}

The two counters are independent monotonic sequences, each with one writer. Because each lane has exactly one writer, there is no merge conflict to resolve within a lane.

\subsection{Invariants}
\label{subsec:reconciliation-invariants}

The correctness of the model rests on three invariants:

\begin{enumerate}
	\item \textbf{The chip never increases its own balance.} Every chip-side mutation is a debit: \texttt{balance -= amount} together with \texttt{debit\_counter += 1}, applied as one atomic write.
	\item \textbf{The server is the only source of credits.} Top-ups, refunds, and operator adjustments mutate the server's balance and increment \texttt{credit\_counter}. The chip picks up these credits passively on the next tap by an online terminal.
	\item \textbf{Counters are monotonic with a single owner each.} \texttt{debit\_counter} only ever grows on the chip; \texttt{credit\_counter} only ever grows on the server.
\end{enumerate}

As long as these three hold, reconciliation reduces to taking the union of two non-conflicting streams.

\subsection{The four operations}
\label{subsec:reconciliation-ops}

\textbf{Top-up.} Always online. The server checks the payment, increments \texttt{balance} and \texttt{credit\_counter}, writes a transaction row, and acknowledges. The chip is not touched and gets reconciled on the next tap.

\textbf{Online debit.} The terminal reads the chip and forwards the request to the server with the chip's current \texttt{debit\_counter} and \texttt{credit\_counter\_seen}. The server applies any pending credits, validates the debit, and returns the new triple \texttt{(balance, debit\_counter, credit\_counter\_seen)}, which the terminal writes atomically to the chip.

\textbf{Offline debit.} The terminal reads the chip, checks the balance, and performs an atomic decrement on the chip. It generates an idempotency key and queues the transaction locally, draining the queue in the background as connectivity allows.

\textbf{Sync.} When a terminal regains connectivity, it pushes queued transactions to the server in batches. The server applies them idempotently using the per-transaction key, updates \texttt{balance}, and advances \texttt{debit\_counter\_seen} to the highest counter value observed.

\subsection{Worked example}
\label{subsec:reconciliation-example}

Consider a top-up and an offline debit happening at the same time. Starting state: chip \texttt{(balance=50, debit\_counter=0, credit\_counter\_seen=0)} and server \texttt{(balance=50, credit\_counter=0, debit\_counter\_seen=0)}.

The user pays 10 tokens at an offline terminal. The chip becomes \texttt{(40, 1, 0)} and the terminal queues a debit of 10. In parallel, a 20-token top-up goes through the server, which becomes \texttt{(70, 1, 0)}.

The user then taps an online terminal. The terminal reads \texttt{(40, 1, 0)} and forwards it to the server. The server sees \texttt{credit\_counter=1 > credit\_counter\_seen=0}, applies the pending 20-token credit, and writes \texttt{(60, 1, 1)} to the chip.

When the offline terminal eventually syncs the queued debit, the server applies it: balance goes from 70 to 60, and \texttt{debit\_counter\_seen} advances from 0 to 1. Both sides converge at \texttt{balance=60} with no conflict resolution because the two streams never collided.

\subsection{Lost transactions}
\label{subsec:reconciliation-lost}

The model does not magically prevent loss. If a terminal dies before draining its queue, the vendor served the customer and the chip subtracted the amount, but the server has no record. The gap is detectable: when \texttt{chip.debit\_counter} exceeds \texttt{debit\_counter\_seen} plus the count of pending queued transactions, the difference represents lost debits.

The policy is to trust the chip's balance immediately, wait a configurable grace period for late syncs, and after that reconcile by writing the chip's balance to the server and logging the discrepancy. The festival absorbs the loss for those transactions; vendors with local receipts can claim manually. The exposure is bounded by the sync interval, an operational risk rather than a correctness bug.

\section{Lessons from related systems and patterns}
\label{sec:why-architecture}

The hybrid model mixes ideas from transit cards, EMV, and the distributed systems literature.

Transit card systems like Suica and Octopus (Section~\ref{subsec:transit-cards}) prove that an on-chip balance works at scale: the balance lives on the chip and the gate does not call any server. Our wristband borrows this core idea, with the same split as Suica (chip-only on spending, back-office on credit).

From commercial festival systems like Tappit and Intellitix (Section~\ref{subsec:campus-cards}), we mainly learned what to avoid: requiring an IT team and on-site servers. Conversations about the Untold deployment confirmed that even with on-site infrastructure, top-up sync delays of two to four hours were common when Wi-Fi got bad. Our architecture sidesteps this entirely.

The EMV offline authorization model (Section~\ref{subsec:emv}) gave us spending limits as a risk control. The event admin could hypotheticly configure maximum transaction amounts and a maximum total offline spend per wristband, as a safety net for the case where someone tampers with a chip, but this should not be doable with proper secured chips.

From the architectural patterns covered in Section~\ref{sec:offline-first}, the relevant takeaways are: \textbf{local-first software}~\cite{Kleppmann2019} does not fit payment systems because conflicting balance updates cannot just be merged; \textbf{CRDTs}~\cite{Shapiro2011} are overkill because lane separation already solves the convergence problem with two integers; \textbf{event sourcing} is the one pattern we use directly, with every transaction recorded as an immutable event for audit and settlement; and \textbf{edge computing} shows up at two levels, the vendor phones running debit logic locally, and the server deployed in a region close to the festivals it serves.

\subsection{Offline operation}
\label{subsec:offline-operation}

During online operation, vendor terminals send each transaction to the server right after it is recorded locally. When the network goes down, the wristband interaction does not change at all. Transactions accumulate in the local queue on the vendor device instead of being sent immediately. Once connectivity returns, the device pushes its queued transactions in a batch. The full convergence mechanics are covered in Section~\ref{sec:reconciliation}.

\subsection{Top-ups: the one thing that needs the network}
\label{subsec:topups}

Topping up is the one operation that cannot happen offline. Real money moves from the attendee's bank account through Stripe into the operator's account, which requires an internet connection. Restricting top-ups to online-only is also what keeps the credit lane fully under server control.

Top-ups happen in two ways. Attendees can top up through the app using Stripe; once the payment clears, the server records the new balance and increments \texttt{credit\_counter}, and the next vendor tap picks up the pending credit. Cash top-up stations at the event submit the credit through the server, which then writes the new balance to the wristband on the spot.

\section{The mobile application}
\label{sec:mobile-app}

There is one mobile app, built in React Native, that attendees and vendors both download. What the user sees depends on their role: attendees see their balance, vendors get a POS terminal. We went with React Native because the same codebase needs to run on regular phones (attendees) and on devices used as POS terminals (vendors) that require NFC support.

\subsection{Attendee view}
\label{subsec:attendee-view}

Attendees see a simple screen showing their token balance, a transaction history, and a way to top up using Stripe. The balance shown in the app is the last known state synced from the server; the real balance during the event lives on the wristband. The app is mostly informational for attendees.

\subsection{Vendor POS mode}
\label{subsec:vendor-pos}

When an admin has whitelisted a device, the app switches to POS mode on that device. The vendor enters the amount, the customer taps their wristband, the app reads the balance, checks it, deducts the amount, writes the new balance back, and logs the transaction. If online, the transaction is sent to the server immediately; otherwise it goes into the local queue.

To become a vendor, the business provides basic information (business name, contact, products). The admin reviews and approves the application, then whitelists specific devices (by device ID) as POS terminals. This whitelisting matters because not just any phone can start accepting payments.

\section{Admin dashboard}
\label{sec:admin-dashboard}

The admin dashboard is a separate web application built in React. Making it a website means any organizer can access it from any browser without downloading anything. The dashboard covers four areas:

\textbf{Event setup.} Create the event, set the token-to-currency rate, define spending limits, configure vendor commissions, and set dates. The event has a lifecycle (setup, active, settlement, closed) and the dashboard controls the transitions.

\textbf{Vendor management.} Review vendor applications, approve or reject them

\textbf{Monitoring and statistics.} Real-time transaction volume and other stats

\section{Backend server}
\label{sec:backend-server}

The backend is built with NestJS, which enforces a clean separation between controllers, services, and repositories, and gives us guards, interceptors, and dependency injection out of the box.

The server is deployed on DigitalOcean, with a serverless Neon PostgreSQL database in the same region. We deploy to a European region to keep latency low for festivals in Romania and the rest of Europe.

The REST API covers four areas: authentication and accounts, event management, vendor management, and sync/reconciliation.

\subsection{Multi-tenancy}
\label{subsec:multi-tenancy}

The server handles all festivals from a single deployment. Every table is scoped by an \texttt{event\_id} column, which acts as the tenant boundary. API calls include the event context (path parameter or token claim), and queries are filtered accordingly. Adding a new festival is just creating a row in the events table.

\subsection{Horizontal scalability}
\label{subsec:horizontal-scalability}

The server is stateless: no session data or cached state in memory. Every request gets what it needs from the database and from the JWT token. We can run as many instances as we want behind a load balancer with no sticky sessions or inter-instance communication. For most festivals a single instance is enough, since most transactions happen on the wristband without touching the server.

\section{Database: PostgreSQL}
\label{sec:database}

The database is PostgreSQL, running as a single instance with the option to add read replicas later. The schema follows the hybrid model directly:

\begin{itemize}
	\item \texttt{users}, \texttt{sessions}, \texttt{accounts}, \texttt{verifications}: identity and auth, managed by Better Auth.
	\item \texttt{events}: per-event configuration (organizer, status, currency, token rate, dates, location).
	\item \texttt{event\_members}, \texttt{vendor\_members}: per-event and per-vendor roles. Authority is held here, not on the global \texttt{users.role} column.
	\item \texttt{vendors}: vendor application per event, with status (\texttt{pending}, \texttt{approved}, \texttt{rejected}, \texttt{suspended}) and commission rate.
	\item \texttt{event\_bracelets}: the wallet, one row per (event, wristband). Holds the canonical \texttt{balance}, \texttt{debit\_counter\_seen}, and \texttt{credit\_counter}, plus the active token (\texttt{tokenIssuedAt}, \texttt{tokenExpiresAt}, \texttt{tokenVersion}) and the assignment status (\texttt{active}, \texttt{revoked}, \texttt{replaced}). A partial unique index keeps at most one active assignment per (event, wristband) and per (event, user).
	\item \texttt{event\_tickets}: admin-issued QR invitations that bind a wristband to a user at the gate.
	\item \texttt{transactions}: one row per money movement, scoped to a bracelet via \texttt{event\_bracelet\_id}. \texttt{type} is \texttt{debit} or \texttt{credit}; \texttt{debit\_counter} or \texttt{credit\_counter} is filled in accordingly. The \texttt{offline} flag and \texttt{idempotency\_key} make sync batches safe to retry.
	\item \texttt{payment\_intents}: provider-agnostic top-up records. On webhook success the credit lands on the exact bracelet referenced by the intent.
	\item \texttt{audit\_logs}: append-only trail of admin actions, indexed by event, user, and entity.
\end{itemize}

Every table that belongs to an event is reachable from the event row, either directly through an \texttt{event\_id} foreign key or through \texttt{event\_bracelets}, which gives us the multi-tenant boundary without a tenant column on every table.

A common worry with using one database for everything is whether it can handle the load. In January 2026, OpenAI published a blog post~\cite{OpenAIPostgres2026} describing how they run ChatGPT, with over 800 million users, on a single primary PostgreSQL on Azure plus around 50 read replicas, processing over a million queries per second at low double-digit millisecond p99 latency. If that setup serves 800 million users, a festival with 50,000 attendees is not what breaks PostgreSQL. Our load is also lighter than a typical web app because most payments happen on the wristband and never touch the database in real time.

If the platform grows to the point where the database becomes a bottleneck, we can follow the same playbook: read replicas, connection pooling (PgBouncer), and table partitioning, all standard techniques that do not require changing application code.
