\chapter{System Architecture}
\label{chap:architecture}

This chapter goes through the architecture of the closed-loop payment system. We start with the big picture, explain why we chose this approach, and then go into each component. The idea is to keep things simple: one server, one app for everyone, and reliable network whenever possible. If the network goes down, the balance cached on the bracelet is what decides whether a payment is approved.

\section{High-level overview}
\label{sec:high-level-overview}

Most existing festival payment systems need a technical team on-site to set up local servers and a private network for the duration of the event. As discussed in Section~\ref{subsec:campus-cards}, festivals like Untold bring dedicated servers to the venue because the balance lives on a remote server and every transaction needs a network round-trip.

Our system takes a different path. We aim to always have reliable internet using Starlink as the primary uplink, paired with a redundant network topology. The wristband-as-cache fallback exists for residual edge cases, but the goal is to never need it. This means there is nothing to deploy at the venue: no local servers, no private network, no on-site IT team. The organizer creates an event from the admin dashboard, configures it, approves vendors, and distributes wristbands. The server runs in the cloud the same way it runs for every other festival on the platform.

The system has four main components: a backend server, a mobile application for attendees and vendors, a web-based admin dashboard, and NFC wristbands that carry the token balance for each attendee. The server is a single multi-tenant service that manages all festivals from one deployment. The mobile app is what attendees and vendors interact with: attendees see their balance and history, vendors use it as a soft point-of-sale terminal. Admins manage the event through a separate web dashboard. The wristband is where the balance actually lives during the event.

\subsection{The hybrid reconciliation model}
\label{subsec:hybrid-overview}

The core architectural decision is how the wristband and the server share responsibility for the balance. The state and the rules that keep the two sides convergent are described in detail in Section~\ref{sec:reconciliation}; the threat model around the chip and the upgrade path to authenticated chips are in Chapter~\ref{chap:security}.

\section{Balance reconciliation model}
\label{sec:reconciliation}

A closed-loop payment system with offline-capable terminals has two writers that can mutate the balance: the server (top-ups) and the chip (vendor payments). During a network partition both can act on their own, and once connectivity is back the system has to converge on a correct final value. Our hybrid model handles this by keeping the two writers in separate lanes that never overlap.

\subsection{State on each side}
\label{subsec:reconciliation-state}

The wristband stores three reconciliation fields per event, plus identity fields written at provisioning:
\begin{itemize}
	\item \texttt{balance}: cached balance in minor units, what the chip can spend from when offline.
	\item \texttt{debit\_counter}: monotonic counter, incremented by one on every chip-side debit. The chip is the only writer.
	\item \texttt{credit\_counter\_seen}: the highest server-side credit sequence number the chip has acknowledged.
	\item identity fields \texttt{bracelet\_uid} and \texttt{event\_id}, written once at provisioning.
\end{itemize}

The server stores the mirror image, per wristband per event:
\begin{itemize}
	\item \texttt{balance}: the canonical balance.
	\item \texttt{credit\_counter}: monotonic counter, incremented by one on every server-side credit. The server is the only writer.
	\item \texttt{debit\_counter\_seen}: the highest \texttt{debit\_counter} value the server has reconciled from any terminal.
\end{itemize}

The two counters are independent monotonic sequences, each with one writer. Because each lane has exactly one writer, there is no merge conflict to resolve within a lane.

\subsection{Invariants}
\label{subsec:reconciliation-invariants}

The correctness of the model rests on three invariants:

\begin{enumerate}
	\item \textbf{The chip never increases its own balance.} Every chip-side mutation is a debit: \texttt{balance -= amount} together with \texttt{debit\_counter += 1}, applied as one atomic write.
	\item \textbf{The server is the only source of credits.} Top-ups, refunds, and operator adjustments mutate the server's balance and increment \texttt{credit\_counter}. The chip picks up these credits passively on the next tap by an online terminal.
	\item \textbf{Counters are monotonic with a single owner each.} \texttt{debit\_counter} only ever grows on the chip; \texttt{credit\_counter} only ever grows on the server.
\end{enumerate}

As long as these three hold, reconciliation reduces to taking the union of two non-conflicting streams.

\subsection{The four operations}
\label{subsec:reconciliation-ops}

\textbf{Top-up.} Always online. The server checks the payment, increments \texttt{balance} and \texttt{credit\_counter}, writes a transaction row, and acknowledges. The chip is not touched and gets reconciled on the next tap.

\textbf{Online debit.} The terminal reads the chip and forwards the request to the server with the chip's current \texttt{balance}, \texttt{debit\_counter}, and \texttt{credit\_counter\_seen}. If the chip's debit counter is ahead of the server, the server trusts the chip as evidence of offline spending that has not synced yet. It reconstructs the spendable balance from the chip state plus any server credits the chip has not seen, caps that value by the server balance, validates the new debit, advances \texttt{debit\_counter\_seen}, and returns the new triple \texttt{(balance, debit\_counter, credit\_counter\_seen)}, which the terminal writes atomically to the chip.

\textbf{Offline debit.} The terminal reads the chip, checks the balance, and performs an atomic decrement on the chip. It generates an idempotency key and queues the transaction locally, draining the queue in the background as connectivity allows.

\textbf{Sync.} When a terminal regains connectivity, it pushes queued transactions to the server in batches. The server applies new counters idempotently using the per-transaction key, updates \texttt{balance}, and advances \texttt{debit\_counter\_seen} to the highest counter value observed. If a queued debit arrives later for a counter that the server has already absorbed through an online chip-aware debit, the server still records the transaction row for vendor history and settlement, but does not subtract the amount again.

\subsection{Worked example}
\label{subsec:reconciliation-example}

Consider a top-up, an offline debit, and then a later online purchase. Starting state: chip \texttt{(balance=50, debit\_counter=0, credit\_counter\_seen=0)} and server \texttt{(balance=50, credit\_counter=0, debit\_counter\_seen=0)}.

The user pays 10 tokens at an offline terminal. The chip becomes \texttt{(40, 1, 0)} and the terminal queues a debit of 10 with counter 1. In parallel, a 20-token top-up goes through the server, which becomes \texttt{(70, 1, 0)}.

The user then pays another 5 tokens at an online terminal. The terminal reads \texttt{(40, 1, 0)} and forwards that chip state with the new debit request. The server sees that the chip is one debit ahead of \texttt{debit\_counter\_seen}, and that the chip has not yet seen credit counter 1. It therefore computes the spendable balance as \texttt{40 + 20 = 60}, capped by the server balance of 70. After the 5-token debit, the server stores \texttt{balance=55} and \texttt{debit\_counter\_seen=2}, and the terminal writes \texttt{(55, 2, 1)} to the chip.

When the offline terminal eventually syncs the queued debit for counter 1, the server does not subtract 10 again, because the online debit already absorbed that missing chip-side spend into the aggregate balance. It inserts the late debit as a historical transaction row for reporting and settlement, marks it as already absorbed, and leaves the balance at 55. Both sides converge without blocking the online purchase on the background sync timing of another vendor device.

\subsection{Lost transactions}
\label{subsec:reconciliation-lost}

The model does not magically prevent loss. If a terminal dies before draining its queue, the vendor served the customer and the chip subtracted the amount, but the server may initially have no itemized transaction row. The online debit path still trusts the chip's lower balance, so the attendee cannot spend the same tokens twice. The missing itemized debit remains detectable as a server transaction-history gap for a counter value below \texttt{debit\_counter\_seen}.

The policy is to trust the chip's balance immediately for authorization, wait a configurable grace period for late syncs to fill in itemized vendor history, and after that log the missing counter range as a discrepancy. The festival absorbs any settlement loss for transactions whose terminal queue never returns; vendors with local receipts can claim manually. The customer balance exposure is bounded because online payments use the chip state when it is ahead of the server.

\section{Lessons from related systems and patterns}
\label{sec:why-architecture}

The hybrid model borrows from three places: transit cards (Section~\ref{subsec:transit-cards}) for the on-chip balance pattern, EMV (Section~\ref{subsec:emv}) for the idea of bounding offline risk through per-card limits, and the offline-first literature (Section~\ref{sec:offline-first}) for what to take and what to leave. From transit we keep the chip-as-spend-cache idea, with credits handled by the back office instead of by the gate. From EMV we keep the framing that offline always trades availability for some risk that has to be capped explicitly. From the patterns in the literature, local-first~\cite{Kleppmann2019} fits as a read cache rather than as the primary copy because money is conserved; CRDTs~\cite{Shapiro2011} are overkill once each balance lane has a single writer; event sourcing is used directly, since every transaction is an immutable row keyed by an idempotency key; and the vendor terminals act as edge nodes~\cite{Shi2016} that run the debit logic locally when the network is gone. The operational lesson from the Untold conversations is the one that pushed us off the on-site-server pattern: even with dedicated infrastructure at the venue, top-up sync delays of two to four hours were common when Wi-Fi degraded.

\subsection{Offline operation}
\label{subsec:offline-operation}

During online operation, vendor terminals send each payment request to the server together with the chip state read from the wristband. When the network goes down, the wristband interaction still succeeds locally: the terminal debits the chip and queues the transaction on the vendor device. Once connectivity returns, the device pushes its queued transactions in a background batch. The full convergence mechanics are covered in Section~\ref{sec:reconciliation}.

\subsection{Top-ups: the one thing that needs the network}
\label{subsec:topups}

Topping up is the one operation that cannot happen offline. Real money moves from the attendee's bank account through Stripe into the operator's account, which requires an internet connection. Restricting top-ups to online-only is also what keeps the credit lane fully under server control.

Top-ups happen in two ways. Attendees can top up through the app using Stripe; once the payment clears, the server records the new balance and increments \texttt{credit\_counter}, and the next vendor tap picks up the pending credit. Cash top-up stations at the event submit the credit through the server, which then writes the new balance to the wristband on the spot.

\section{The mobile application}
\label{sec:mobile-app}

There is one mobile app, built in React Native, that attendees and vendors both download. What the user sees depends on their role: attendees see their balance, vendors get a POS terminal. We went with React Native because the same codebase needs to run on regular phones (attendees) and on devices used as POS terminals (vendors) that require NFC support.

\subsection{Attendee view}
\label{subsec:attendee-view}

Attendees see a simple screen showing their token balance, a transaction history, and a way to top up using Stripe. The balance shown in the app is the last known state synced from the server; the real balance during the event lives on the wristband. The app is mostly informational for attendees.

\subsection{Vendor POS mode}
\label{subsec:vendor-pos}

When an admin has whitelisted a device, the app switches to POS mode on that device. The vendor enters the amount and the customer taps their wristband. If online, the app reads the chip state, sends it to the server with the amount, and writes the server's returned state back to the chip. If offline, the app checks the chip balance locally, deducts the amount, increments the chip debit counter, writes the new state back, and stores the debit in the local queue for background sync.

To become a vendor, the business provides basic information (business name, contact, products). The admin reviews and approves the application, then whitelists specific devices (by device ID) as POS terminals. This whitelisting matters because not just any phone can start accepting payments.

\section{Admin dashboard}
\label{sec:admin-dashboard}

The admin dashboard is a separate web application built in React. Making it a website means any organizer can access it from any browser without downloading anything. The dashboard covers four areas:

\textbf{Event setup.} Create the event, set the token-to-currency rate, define spending limits, configure vendor commissions, and set dates. The event has a lifecycle (setup, active, settlement, closed) and the dashboard controls the transitions.

\textbf{Vendor management.} Review vendor applications, approve or reject them

\textbf{Monitoring and statistics.} Real-time transaction volume and other stats

\section{Backend server}
\label{sec:backend-server}

The backend is a single multi-tenant service. It exposes four families of endpoints (authentication and accounts, event management, vendor management, and sync/reconciliation), is stateless across instances, and runs behind a managed load balancer. The choice of framework, the database, and the deployment target are implementation details that belong to the next chapter (Section~\ref{subsec:impl-deployment}).

\subsection{Multi-tenancy}
\label{subsec:multi-tenancy}

The server handles all festivals from a single deployment. Every table that belongs to an event is reachable from the event row, either directly through an \texttt{event\_id} foreign key or through \texttt{event\_bracelets}. API calls include the event context (path parameter or token claim) and queries are filtered accordingly. Adding a new festival is just creating a row in the events table.

\subsection{Horizontal scalability}
\label{subsec:horizontal-scalability}

The server is stateless: no session data or cached state in memory. Every request gets what it needs from the database and from the bearer token sent by the client. The service can run as many instances as needed behind the load balancer with no sticky sessions or inter-instance communication. For most festivals a single instance is enough, since the majority of payments happen on the wristband without touching the server in real time.

\section{Database}
\label{sec:database}

The data model follows the hybrid reconciliation model directly. The canonical balance and the two reconciliation counters live on \texttt{event\_bracelets}, every money movement is logged in \texttt{transactions} with the appropriate counter set, and top-up records sit in \texttt{payment\_intents}. The full schema (table list, column purposes, constraints, and the safety nets that Postgres enforces directly) is described in Chapter~\ref{chap:implementation}, Section~\ref{sec:impl-schema}.

The choice of a single relational database rather than a sharded or distributed setup is conservative on purpose. Most transactions never touch the database in real time, because they happen on the wristband during normal vendor flow; the database mostly absorbs sync batches and dashboard queries. A recent published reference point~\cite{OpenAIPostgres2026} from OpenAI describes ChatGPT's user-facing workload, with hundreds of millions of users, running on a single primary Postgres with read replicas at low double-digit millisecond p99 latency, which suggests that a festival's volume is well inside what a single relational database handles. If load ever justifies it, the standard scaling path (read replicas, connection pooling, table partitioning) does not require changes to application code.
